# Cold-Path RLS P0 Implementation Plan

**Goal:** Enforce PostgreSQL row-level security on every buffered cold or hot+cold tuple without regressing the native hot-only scan path.

**Architecture:** Keep the existing native child fast path unchanged because PostgreSQL already qualifies those heap tuples. For buffered merge output, materialize the minimally required base-table attributes into `ss_ScanTupleSlot` and return them through PostgreSQL `ExecScan`, which applies the already-compiled `plan.qual` (including RLS security quals) and normal projection. Use PostgreSQL's generic Var walker so custom policy expressions receive every referenced base attribute without attempting to reinterpret policy semantics in Rust.

**Tech stack:** Rust, pgrx/PostgreSQL CustomScan executor APIs, Tokio PostgreSQL E2E harness, Parquet-backed cold storage.

### Task 2: Apply PostgreSQL quals to buffered tuples

**Files:**

- Modify: `crates/pg_koldstore/src/merge_scan/pg.rs`
- Modify: `crates/pg_koldstore/src/merge_scan/pg/qual.rs`
- Modify: `crates/pg_koldstore/src/merge_scan/pg/emit.rs`
- Modify: `crates/pg_koldstore/src/merge_scan/pg/tuple.rs`
- Modify as required by the final shape: `crates/pg_koldstore/src/merge_scan/pg/hot.rs`

1. Collect target-list and qual Var attnums with PostgreSQL `pull_varattnos`.
2. Keep Parquet/SPI projection limited to the attributes needed for output, merge filtering, and qual evaluation; map only referenced physical attributes so dropped slots stay NULL.
3. Store buffered row images in base-table tuple layout.
4. Use `ExecScan` with a buffer access callback so PostgreSQL evaluates `ps.qual` and projection for each emitted tuple.
5. Keep native hot-child streaming unchanged to avoid duplicate qualification and preserve its performance. For buffered scans, read complete hot winners under the relation owner with `SECURITY_NOFORCE_RLS`; push only primary-key predicates before merge.
6. Restrict cold PK pruning to scalar types whose PostgreSQL semantics match stored stats, and reject nondeterministic PK collations before management because byte-distinct collation-equal values cannot safely form separate Rust merge keys.
7. Run the focused E2E and confirm all hot/cold phases pass.
